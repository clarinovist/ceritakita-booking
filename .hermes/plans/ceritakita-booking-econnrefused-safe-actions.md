# CeritaKita Booking ECONNREFUSED ::1/127.0.0.1:80 Safe Action Plan

Date: 2026-05-22
Scope: ceritakita-booking on VPS nugrohopramono
Constraint: app build/deploy/recreate must go through CI/CD, not direct VPS build/recreate.

## What was executed safely

1. Reverted unsafe local docker-compose change that had added host port mapping `80:3000`.
   - Current local compose only maps `3001:3000`.
   - Remote deployed compose also only maps `3001:3000`.

2. Inspected CI/CD workflow.
   - `.github/workflows/build.yml` builds/pushes image on push to `main`.
   - Deploy job SSHes to VPS, runs `git reset --hard origin/main`, `docker compose pull`, `docker compose up -d`.
   - Therefore repo/app changes should be merged to `main` and deployed by GitHub Actions.

3. Inspected VPS nginx config read-only.
   - `/etc/nginx/sites-enabled/ceritakitastudio.site` upstream is correct:
     `server 127.0.0.1:3001;`
   - No nginx misrouting to `127.0.0.1:80` for this site.

4. Inspected VPS error monitoring read-only.
   - Alerts are generated by `error-monitor.service` running `/root/error_monitor.js`.
   - It follows `docker logs --since 5m -f ceritakita-booking` and sends Telegram alert when log lines match broad error regexes.

## Root cause found

The app/container itself is healthy and listening on port 3000. The alert is triggered by log lines from Next.js when a request tries to proxy cloud metadata URLs:

- `Failed to proxy http:/169.254.169.254/metadata/v1.json AggregateError [ECONNREFUSED]`
- `Failed to proxy http:/169.254.169.254/latest/meta-data/ AggregateError [ECONNREFUSED]`
- `Failed to proxy http:/metadata.google.internal/computeMetadata/v1/instance/?recursive=true AggregateError [ECONNREFUSED]`

Those contain nested errors:

- `Error: connect ECONNREFUSED ::1:80`
- `Error: connect ECONNREFUSED 127.0.0.1:80`

This pattern is consistent with external/bot SSRF-style probes against Next image/proxy behavior, not the app needing to listen on port 80.

## What not to do

- Do not add `80:3000` to docker-compose.
- Do not change app `PORT` to 80.
- Do not add `NET_ADMIN`/iptables redirect inside the container.
- Do not direct build/recreate on the VPS.

## Recommended next fix options

### Option A — monitor-only fix (lowest app risk, direct VPS service change)
Update `/root/error_monitor.js` ignore patterns to suppress known benign metadata probe log lines, then restart only `error-monitor.service`.

Pros:
- No app/container deploy.
- No impact to nginx or existing containers.
- Stops Telegram false positives.

Cons:
- Direct VPS config/service change, not CI/CD unless we first move this monitor script into a repo-managed deployment path.

Suggested ignore patterns:

```js
/Failed to proxy http:\/?169\.254\.169\.254\//i,
/Failed to proxy http:\/?metadata\.google\.internal\//i,
/connect ECONNREFUSED (::1|127\.0\.0\.1):80/i,
```

Only use the broad `connect ECONNREFUSED ...:80` pattern for ceritakita if possible; otherwise it could hide real alerts in other apps.

### Option B — app/repo fix via CI/CD (preferred if we want infra-as-code)
Add a repo-managed guard for metadata/private-host requests if applicable, then merge to main and let GitHub Actions deploy.

Need more validation because Next's built-in `/_next/image` handling may happen before app middleware depending on version/config.

### Option C — nginx-level block (effective but affects nginx)
Block requests to `/_next/image` where the `url` query points to private metadata hosts. Requires `nginx -t` and reload. This is safe when targeted to the ceritakita server block, but nginx is shared infrastructure, so should require explicit approval.
